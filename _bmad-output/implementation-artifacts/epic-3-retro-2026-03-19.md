## Successes & Strengths
- **Background Task Orchestration:** Successful use of `anyio.to_thread.run_sync` for computationally heavy tasks like extraction and embedding, avoiding event loop blocking.
- **Thorough Error Handling:** Introduced `error_message` in the `Document` schema to track processing failures accurately and present them gracefully on the mobile app.
- **Licensing Awareness:** Opted for `pypdf` early instead of `PyMuPDF` to avoid AGPL licensing issues without sacrificing core functionality.
- **UI State Management:** Built a robust, accessible processing pipeline UI in Flutter using a 3-second polling mechanism with careful cleanup and `Reduce Motion` support.

## Challenges & Growth Areas
- **Strict Schema Constraints:** `Document.page_count` was strictly non-nullable causing friction during file upload since processing isn't synchronous. Solved by defaulting to `0` and updating asynchronously.
- **Multipart Upload Restrictions:** `Dio` interceptors were forcing `application/json` globally, requiring refactoring of auth injection to support `multipart/form-data`.
- **Backend Schema Evolution:** The UI needed `file_size`, `page_count`, and `error_message` from the backend `DocumentPublic` response, requiring mid-sprint additions to the FastAPI schemas and tests.

## Key Insights & Learnings
- **Isolated Boundaries:** Keeping integrations like `boto3` limited to `StorageService` proved invaluable.
- **Mocking Extraneous Services:** Using injectable/monkeypatch architectures permitted extensive unit testing without needing AWS credentials or live `SentenceTransformers` execution.
